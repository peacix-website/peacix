# 🔒 .gitignore Configuration Guide

## Overview
This project uses comprehensive `.gitignore` files to prevent sensitive and unnecessary files from being committed to Git.

---

## 📁 Location of .gitignore Files

### 1. **Root Level** (`/peacix/.gitignore`)
- Covers the entire monorepo
- Prevents committing root-level dependencies, build outputs, and secrets

### 2. **Frontend Level** (`/peacix/frontend/.gitignore`)
- Specific to React/Vite frontend
- Prevents committing node_modules, dist, env files, etc.

---

## 🚫 What's Ignored (And Why)

### **Environment Variables** ⚠️ CRITICAL
```
.env
.env.local
.env.development.local
.env.test.local
.env.production.local
```
**Why:** Contains API keys, database URLs, and other secrets that should NEVER be public.

**Exception:** `!.env.example` - Template file with placeholder values is safe to commit.

---

### **Dependencies**
```
node_modules/
package-lock.json
yarn.lock
pnpm-lock.yaml
```
**Why:** 
- `node_modules/` is huge and regenerated by `npm install`
- Lock files can be platform-specific (though some teams commit these)

---

### **Build Outputs**
```
dist/
build/
out/
.next/
```
**Why:** Generated files that should be built fresh for each deployment.

---

### **Logs & Debug Files**
```
*.log
npm-debug.log*
yarn-debug.log*
coverage/
.nyc_output/
```
**Why:** Temporary debugging information, not needed in version control.

---

### **OS Files**
```
.DS_Store (macOS)
Thumbs.db (Windows)
._* (macOS metadata)
```
**Why:** System-generated files that clutter the repo.

---

### **IDE Files**
```
.vscode/
.idea/
*.swp
*.swo
*~
```
**Why:** Editor-specific settings that vary by developer.

**Exception:** You can commit `.vscode/settings.json` if you want team-wide editor settings.

---

### **Supabase**
```
.supabase/
functions/**/*.map
secrets.json
```
**Why:** 
- Local Supabase CLI config
- Source maps for edge functions
- Secret credentials

---

### **Security Files**
```
*.pem
*.key
*.crt
credentials.json
secrets.json
```
**Why:** SSL certificates, private keys, and credentials should NEVER be committed.

---

### **Database Files**
```
*.sql
backups/
```
**Why:** Database dumps can contain sensitive user data.

**Exception:** `!*.sql.example` - Schema templates without data are OK.

---

### **Cloud Deployment**
```
.vercel/
.netlify/
.aws/
```
**Why:** Platform-specific configs often contain secrets or auto-generated files.

---

## ✅ What SHOULD Be Committed

### Safe to Commit:
- ✅ Source code (`.jsx`, `.js`, `.ts`, `.tsx`)
- ✅ CSS files (`.css`, `.scss`)
- ✅ HTML files
- ✅ Public assets (images, fonts)
- ✅ `package.json` (dependencies list)
- ✅ `.env.example` (template with placeholders)
- ✅ Documentation (`.md` files)
- ✅ Config files (without secrets)

### Example `.env.example`:
```bash
# Copy this to .env and fill in your actual values
VITE_SUPABASE_URL=your_supabase_url_here
VITE_SUPABASE_ANON_KEY=your_anon_key_here
VITE_RAZORPAY_KEY_ID=your_test_key_here
VITE_ENVIRONMENT=development
```

## 🔍 How to Check What's Ignored

### See what Git is ignoring:
```bash
git status --ignored
```

### Check if a specific file is ignored:
```bash
git check-ignore -v path/to/file
```

### Force add a file that's ignored (NOT RECOMMENDED):
```bash
git add -f path/to/file
```

## ⚠️ Common Mistakes

### ❌ BAD: Committing .env file
```bash
git add .env  # NEVER DO THIS!
```

### ❌ BAD: Removing .gitignore after committing secrets
If you accidentally committed secrets:
1. Remove the file from Git history
2. Rotate the compromised credentials
3. Add to .gitignore

### ✅ GOOD: Using .env.example
```bash
cp .env.example .env
# Edit .env with actual values (never commit!)
```

---

## 🛡️ Security Best Practices

### 1. **Check Git History for Accidental Commits**
```bash
git log --all --full-history -- "**/*.env"
git log --all --full-history -- "**/secrets.json"
```

### 2. **Remove Sensitive Files from History**
If you accidentally committed secrets:
```bash
# Remove from last commit
git reset HEAD~1
# Or use BFG Repo-Cleaner for deeper history
```

### 3. **Rotate Compromised Credentials**
If secrets were committed:
1. Delete them from all branches
2. Generate new API keys
3. Update all environments

## 📋 Checklist Before First Commit

- [ ] `.env` files are NOT in the repository
- [ ] `node_modules/` is ignored
- [ ] No `.pem`, `.key`, or `.crt` files
- [ ] No hardcoded passwords or API keys in source code
- [ ] Database credentials are in `.env` only
- [ ] `.gitignore` files are committed

## 🔧 Troubleshooting

### File still shows up in `git status` after adding to `.gitignore`?

The file was already tracked. Remove it first:
```bash
git rm --cached path/to/file
git commit -m "Remove sensitive file from tracking"
```

### Want to see what would be ignored?
```bash
git clean -ndX  # Shows what would be deleted
```

### Need to commit a lock file despite .gitignore?
```bash
git add -f package-lock.json
git commit -m "Add lock file for CI"
```

## 🎯 Quick Reference

| File Type | Ignore? | Reason |
|-----------|---------|--------|
| `.env` | ✅ YES | Contains secrets |
| `node_modules/` | ✅ YES | Auto-generated |
| `dist/` | ✅ YES | Build output |
| `package.json` | ❌ NO | Needed for deps |
| `.env.example` | ❌ NO | Template only |
| `src/App.jsx` | ❌ NO | Source code |
| `*.log` | ✅ YES | Temporary |
| `.DS_Store` | ✅ YES | OS file |

---

**Remember:** When in doubt, ask yourself: "Would someone else need this file to run the project?" If yes, commit it. If no (or if it contains secrets), ignore it!
